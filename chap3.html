第3章 WWWの基本要素とその発展

技術のつながりを理解する旅

Webの礎を築いた先駆者たちの試行錯誤をたどり、現代技術へと至る発展の歴史を深掘りします。

World Wide Webの始まり

発端: 1989年、CERNの物理学者ティム・バーナーズ＝リーが、研究者間の情報共有を効率化するために「Information Management: A Proposal」を執筆。

着想: 「地球上のあらゆるコンピュータの情報がリンクされ、誰もがアクセスできる情報空間」というアイデア。これがWWWの原型となります。

実現: 1990年、彼は世界初のWebサーバ、Webブラウザ兼エディタを開発し、最初のWebサイトを公開しました。

WWWを構成する3つの基本仕様

バーナーズ＝リーは、特定のソフトウェアではなく、誰でも実装できる「仕様」を中心にWWWを設計しました。これが技術の発展と普及を促しました。

Webブラウザ

① URI

(情報の「住所」)

② HTTP

(情報の「受渡方法」)

リクエスト →

← レスポンス

Webサーバ

③ HTML

(情報の「表現方法」)

流れ: ブラウザが ① URI を使って、サーバに ② HTTP でリクエストを送信。サーバはレスポンスとして ③ HTML を返し、ブラウザがそれを解釈して画面に表示します。

URI: 理想と現実の狭間で

先行技術の理想: ザナドゥ

テッド・ネルソンが構想した「ザナドゥ」では、リンクは双方向で、バージョン管理や著作権管理まで統合した非常に高度なものでした。

実現が非常に複雑で、大規模な分散システムには不向きでした。

WWWの現実的な選択: 単純化

WWWは実現可能性を重視し、リンクを一方向の単純なものにしました。これにより爆発的な普及が可能になりました。

副作用として、参照先が消える「リンク切れ」や内容の変質といった問題が生まれました。

HTTP: シンプルさと柔軟性

既存プロトコルよりも高速な応答を目指し、シンプルに設計。さらに、既存システムとの共存のために2つの工夫がされました。

1. スキーム

URIの先頭部分 (`http:`, `ftp:`) でプロトコルを指定可能に。これにより、FTPなど既存のサーバで公開されている情報もWWWの枠組みで参照できるようになりました。

ftp://ftp.example.com/file.zip

2. フォーマットネゴシエーション

クライアントが「自身が解釈できる形式」をサーバに通知。サーバは必要に応じて形式を変換して応答できました。これによりHTML以外の形式も扱える柔軟性を持ちました。

HTTPを体験する：telnet通信実験

ブラウザが裏側で行っているHTTP通信を、手元のコマンドで再現してみましょう。これにより、HTTPが人間にも読めるシンプルなテキストのやり取りであることがわかります。

1. サーバーへの接続

まずWebサーバー（info.cern.ch）の80番ポートに接続します。

# ターミナルで実行
telnet info.cern.ch 80

成功すると、次のようなメッセージが表示されます。

Trying 188.184.100.182...
Connected to info.cern.ch.
Escape character is '^]'.

2. HTTPリクエストの送信

接続後、以下の2行を入力し、最後にEnterキーを2回押して空行を送信します。

GET /hypertext/WWW/TheProject.html HTTP/1.1
Host: info.cern.ch

3. HTTPレスポンスの受信

サーバーから応答が返ってきます。これがブラウザが受け取る情報の「生」の状態です。

HTTP/1.1 200 OK
Date: Tue, 25 Jun 2024 03:44:00 GMT
Server: Apache
Last-Modified: Thu, 03 Dec 1992 08:37:20 GMT
Content-Type: text/html
... (レスポンスヘッダ) ...

<HEADER>
<TITLE>The World Wide Web project</TITLE>
<NEXTID N="55">
</HEADER>
<BODY>
... (HTML本体) ...

HTMLの源流: SGML

HTMLは、国際標準のマークアップ言語「SGML」を元に作られました。

SGMLの重要な思想: 構造と体裁の分離

文書の論理構造

<title>題名</title>
<author>著者</author>

文書の「意味」をタグで表現

↔

文書の体裁

中央寄せ
文字サイズ: 16px

見た目の情報は表示側が決定

この分離により、特定のソフトウェアに依存せず、多様なデバイスで文書を表示できる可能性が生まれました。

HTMLの誕生: SGMLの簡略化

SGMLは非常に高機能でしたが、その代償として仕様が複雑でした。特に、文書ごとに構造定義(DTD)を作成する必要があり、手軽な情報発信には不向きでした。

そこでバーナーズ＝リーは、SGMLの厳密さを捨て、WWWに必要十分な機能に絞ったHTMLを考案しました。

  • DTDを毎回定義する必要がない。
  • SGMLのサブセットではなく「親戚のように見える」が、処理を簡略化できる別物として設計。
  • 当初からあった「論理構造と体裁の分離」思想により、テキストベースのブラウザでも閲覧が可能だった。

CSSの登場: 思想への回帰

混乱の時代: ブラウザ戦争

WWWの商業利用が進むと、NetscapeとInternet Explorerがシェアを争い、見た目をリッチにするための独自拡張を乱発しました。

<font color="red">...</font>
<center>...</center>

結果、HTMLに見た目の情報が混ざり、「構造と体裁の分離」という当初の思想が失われてしまいました。

思想への回帰: CSS

この混乱を収拾するため、W3Cは1996年にCSS (Cascading Style Sheets) を策定。HTMLから視覚情報を再び分離する道筋を示しました。

/* CSSで見た目を定義 */
h1 { color: red; text-align: center; }

HTMLは再び構造に専念し、CSSが見た目を担当することで、保守性や再利用性が大幅に向上しました。

XML: データ交換という新たな要請

WWWが普及すると、人間が見る文書だけでなく、プログラム間でデータを交換したいという需要が高まりました。

なぜHTMLではダメなのか？

HTMLは人間が見るための言語であり、同じ内容でも多様な書き方が可能です。この「あいまいさ」がプログラムによる自動処理を困難にしていました。

XMLの誕生

そこで、SGMLからデータ記述に特化したサブセットとしてXML (Extensible Markup Language) が1998年に誕生。タグを自由に定義でき、スキーマでデータ構造を厳密に規定できるため、データ交換に最適でした。

<book>
  <title>Webを支える技術</title>
  <author>山本 陽平</author>
</book>

XHTML: 理想主義の挑戦と衰退

XMLの登場後、「文書もデータの一種なのだから、HTMLもXMLの厳格なルールで再定義しよう」という動きが起こりました。これがXHTMLです。

目指した理想

Web上のあらゆる情報をXMLベースで統一的に扱う世界。

現場の現実

表現力は変わらないのに、終了タグの必須化など、記述が厳格になるだけで制作者のメリットが少なく、受け入れられませんでした。

結果、W3CはXHTMLの開発を2009年に停止。理想の追求が必ずしも良い結果を生むとは限らない教訓を残しました。

HTML5からLiving Standardへ

XHTMLが行き詰まる一方、現場ではWebアプリケーションの進化に伴い、より豊かなHTML表現が求められていました。

1

2004年: WHATWGの発足

W3Cの進め方に不満を持ったブラウザベンダー(Apple, Mozilla, Opera)が、現場のニーズを反映させるためWHATWGを結成。

2

2014年: HTML5勧告

WHATWGの草案を取り入れ、W3CがHTML5を勧告。<video>や<canvas>などが追加され、表現力が大幅に向上。

3

2019年: Living Standardへ統一

その後も仕様策定はW3CとWHATWGで分裂していましたが、最終的にWHATWGが策定する「バージョン番号のない、常に進化し続ける仕様」であるHTML Living Standardに一本化されました。

第3章のまとめ

WWWの誕生: 実現可能性を重視したティム・バーナーズ＝リーの現実的な設計思想（単純化・柔軟性）が、爆発的な普及の鍵だった。

HTMLの進化: SGMLの「構造と体裁の分離」思想を継承。ブラウザ戦争で一時失われたが、CSSの登場で回復。データ交換の要請からXMLが生まれた。

仕様策定の主役交代: 理想を追ったXHTMLは衰退し、開発現場のニーズを汲んだWHATWGが主導権を握り、HTMLは常に進化するLiving Standardとなった。

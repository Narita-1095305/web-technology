第6章 従来型のWebアプリケーション

状態管理と画面遷移の仕組みを学ぶ

Go言語によるToDoアプリ開発を通して、サーバーサイドでHTMLを生成する伝統的なWebアプリケーションの構造を深く理解します。

Goによる簡単なWebサーバ作成

① 文字列を返すサーバ

Goの標準ライブラリ `net/http` を使うと、非常に短いコードでHTTPサーバを起動できます。`http.HandleFunc`でパスと処理関数を紐付け、`http.ListenAndServe`でサーバを開始します。

package main
import ("fmt"; "log"; "net/http")

// "/hello"へのリクエストを処理するハンドラ
func hello(w http.ResponseWriter, r *http.Request) {
    fmt.Fprint(w, "Hello, Web application!")
}

func main() {
    http.HandleFunc("/hello", hello) // URLパスと関数を紐付け
    log.Fatal(http.ListenAndServe(":8080", nil)) // 8080ポートで待受
}

② 静的ファイルを返すサーバ

`http.FileServer` を使うと、指定したディレクトリ以下の静的ファイル（HTML, CSS等）を配信するサーバを簡単に実装できます。

func main() {
    // "/static/"以下のパスへのリクエストを、"static"ディレクトリのファイルにマッピング
    fs := http.FileServer(http.Dir("static"))
    http.Handle("/static/", http.StripPrefix("/static/", fs))
    
    log.Fatal(http.ListenAndServe(":8080", nil))
}

ToDoアプリの基本構造

簡単なToDoアプリを題材に、Webアプリの基本を学びます。まずは固定のリストを表示する機能から。

Goプログラム (main.go)

`html/template` パッケージを使い、HTMLテンプレートと動的なデータを組み合わせてレスポンスを生成します。

var todoList []string // ToDoを保持する変数

// /todo へのリクエストを処理
func handleTodo(w http.ResponseWriter, r *http.Request) {
    // テンプレートファイルをパース
    t, _ := template.ParseFiles("templates/todo.html")
    // テンプレートにtodoListを埋め込んでレスポンスを生成
    t.Execute(w, todoList)
}

func main() {
    todoList = append(todoList, "顔を洗う", "朝食を食べる")
    http.HandleFunc("/todo", handleTodo)
    // ...サーバ起動
}

HTMLテンプレート (templates/todo.html)

`{{range .}}` と `{{end}}` で渡されたデータ（スライス）をループ処理し、リスト項目を動的に生成します。

<ul class="todo-list">
    {{range .}} <!-- ループ開始 -->
    <li class="todo-item">
        <label>{{.}}</label> <!-- 各要素を表示 -->
    </li>
    {{end}} <!-- ループ終了 -->
</ul>

画面遷移の問題点

ToDo項目を追加する機能を実装します。しかし、単純な実装では問題が発生します。

問題: POST後のリロード

フォームからPOSTリクエストでデータを追加した後、ブラウザをリロードすると...
→ 同じPOSTリクエストが再送信され、データが二重に登録されてしまいます。

原因

ブラウザは最後に表示したページのURL（この場合 `/add`）をリロードしようとするため。

通信フロー

① /addへPOST → ② 項目追加後のHTMLをレスポンス → ③ リロードで再び/addへPOST

POST /add

リロードで同じ処理が繰り返される

解決策: Post-Redirect-Get (PRG)

この問題を解決する定番パターンが「Post-Redirect-Get」です。

PRGパターンの動作

POSTリクエストでデータ処理後、HTMLを直接返さず、303 See Otherというステータスコードと一覧ページのURL (`/todo`) を持つリダイレクトレスポンスを返します。

ブラウザは指示に従い、一覧ページにGETリクエストを再送信します。

通信フロー

① POST /add

② Redirect to /todo (サーバからの指示)

③ GET /todo (ブラウザが再リクエスト)

この結果、ブラウザのURLは `/todo` になり、リロードしても安全なGETリクエストが送信されるだけになります。

Goによる実装

`http.Redirect`関数を使うだけで簡単に実装できます。

func handleAdd(w http.ResponseWriter, r *http.Request) {
    // ... ToDo項目追加処理 ...
    
    // /todo へリダイレクト
    http.Redirect(w, r, "/todo", http.StatusSeeOther)
}

状態管理とステートレスなHTTP

現在のToDoアプリは、誰がアクセスしても同じリストが表示されます。これはHTTPがステートレス（状態を持たない）プロトコルだからです。

問題点

サーバーは個々のリクエストを独立したものとして扱うため、「前のリクエストと同じ人」ということを判断できません。

必要なこと

ユーザごとにToDoリストを管理するには、リクエストごとにユーザを識別し、状態を管理する仕組みが必要です。

解決策: Cookieとセッション

ステートレスなHTTP上で状態を管理するための仕組みが、Cookieとセッションです。

①初回アクセス

②サーバがIDを発行

③レスポンスでCookieを送信

Set-Cookie: sessionID=x123

④以降リクエストでCookieを送信

Cookie: sessionID=x123

サーバはセッションIDを使ってユーザを識別し、ユーザごとの情報（セッション情報）をサーバ側で管理します。

Cookieの詳細仕様

`Set-Cookie`ヘッダには、値だけでなく様々な属性を指定して、その挙動を制御できます。

Set-Cookie: name=value; Expires=...; Domain=...; Path=...; Secure; HttpOnly; SameSite=...
Expires/Max-Age: 有効期限。指定がなければブラウザを閉じると消えるセッションクッキーになる。
Domain: Cookieを送信する対象のドメイン。
Path: Cookieを送信する対象のパス。
Secure: HTTPS通信でのみCookieを送信する。
HttpOnly: JavaScriptからのアクセスを禁止し、XSS攻撃のリスクを低減。
SameSite: CSRF攻撃対策。別ドメインからのリクエストでCookieを送信するかを制御 (Strict/Lax/None)。

セキュリティのため、特にSecure, HttpOnly, SameSite属性を適切に設定することが極めて重要です。

ユーザ認証の実装

セッションIDだけではブラウザを識別するのみ。ユーザ自身を識別するには、ユーザIDとパスワードによる認証が必要です。

①ログイン成功

②セッション情報にユーザIDを紐付ける

④ユーザごとの情報をDBから取得

③以降のアクセスでセッションからユーザIDを特定

パスワードの安全な保存

パスワードを平文で保存するのは非常に危険です。サーバ側では、パスワードから計算したハッシュ値を保存し、ログイン時には入力されたパスワードのハッシュ値と比較して認証を行います。

// 登録時: パスワードからハッシュ値を計算して保存
hashedPassword, _ := bcrypt.GenerateFromPassword(password, ...)

// 認証時: 入力されたパスワードと保存されたハッシュ値を比較
err := bcrypt.CompareHashAndPassword(hashedPassword, password)

Webアプリフレームワークの役割

ここまで見てきたように、実用的なWebアプリを作るには、本質的な機能以外に多くの定型的な処理（ルーティング、セッション管理、認証など）が必要です。

フレームワーク = 複雑さの隠蔽

Webアプリケーションフレームワークは、これらの複雑な定型処理を肩代わりし、開発者がアプリケーションのコア機能の開発に集中できるようにする「骨組み」です。

フレームワークを使いこなすためにも、その裏側で動いている本章のような仕組みを理解しておくことが、問題解決能力や応用力に繋がります。

第6章のまとめ

画面遷移: データの登録・更新など状態を変更する処理の後は、Post-Redirect-Get (PRG) パターンで意図しない再実行を防ぐのが定石。

状態管理: ステートレスなHTTPで状態を扱うため、Cookieを使ってクライアントとサーバ間でセッションIDをやり取りし、ユーザを識別する。

ユーザ認証: ログイン機能によりセッションとユーザを紐付け、ユーザごとのデータ管理を実現。パスワードはハッシュ化して安全に保存する。
